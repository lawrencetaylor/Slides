What do we mean by a "booking"?

The concept of a "booking" is not well-defined.

At best, we can provide a model which satisfies a specific client's requirements at a certain point in time.

It's a concept tied to the business.
Different people have different views on what a booking is
Financial aspects of the booking: only concerned about the numbers
Recommendation engine: only concerned with what is booked
Cusomer services require a more complete model.
Any attempt to distill the notion of a booking in to a type is flawed - we can never satisfy future requirements, let alone attempt to understand todays.
But we should endeavor to build software that embraces change.

Software design principles

Aim to design for, and encapsulate volatility (Single Responsibility Principal).

We should be able to handle an evolving domain and changes in requirements without having to "re-design".

We can't plan for the unknowable - but we know that there is volatility is how clients use and interpret the booking concept.

This is precisely what event sourcing allows us to do: design a system which encapsulates this volatility.

let us define an entity by the events that it experiences
